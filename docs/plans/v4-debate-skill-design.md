# v4: debate 스킬 설계

## 개요

범용 AI 토론 엔진. 여러 AI 모델(Claude, Gemini, Codex)이 주어진 주제에 대해 라운드테이블 토론을 벌이고, 결론과 토론 기록을 구조화된 노트로 저장한다.

## 핵심 결정 사항

| 항목 | 결정 |
|------|------|
| 목적 | 범용 토론 엔진 (어떤 주제든 다각도 토론 → 결론 도출) |
| 토론 형식 | 라운드테이블 기본 + 주제 성격에 따라 자동 선택 (디베이트, 소크라테스) |
| 참여자 | 기본 3인 (Claude + Gemini + Codex), 커스텀 가능 |
| Claude 역할 | 참여자 겸 종합자 (토론 참여 + 최종 종합) |
| 종료 조건 | 수렴 기반 + 주제 복잡도에 따른 동적 max round |
| 호출 방식 | omc-teams (tmux 워커) |
| 결과물 | 별도 폴더 `insights/debates/`, 두괄식 (결론 → 근거 → 토론 기록) |
| 스킬 이름 | `debate` |

## Step 1: 주제 분석

Claude가 주제를 받으면 다음을 판단한다.

### 복잡도 레벨 (1-5)

| 레벨 | 기준 | max round | 예시 |
|------|------|-----------|------|
| 1 | 단순 사실/의견 | 2 | "탭 vs 스페이스" |
| 2 | 양면성 있는 주제 | 3 | "모노레포 vs 멀티레포" |
| 3 | 다각도 분석 필요 | 3 | "AI가 일자리를 대체할 것인가" |
| 4 | 전문 지식 + 맥락 필요 | 4 | "한국 AI 규제 방향성" |
| 5 | 철학적/근본적 논쟁 | 5 | "AGI는 의식을 가질 수 있는가" |

### 토론 형식 자동 선택

| 형식 | 조건 | 설명 |
|------|------|------|
| 라운드테이블 | 기본값 | 자유 의견 → 상호 반응 → 수렴 |
| 디베이트 | 명확한 찬반 가능 시 | 찬성/반대 배정 → 반론 → 재반론 |
| 소크라테스 | "왜", "어떻게" 류 질문 | 질문자 1 + 답변자들 |

### 참여자

- 기본값: Claude + Gemini + Codex (3인)
- `--models` 옵션으로 커스텀 가능

## Step 2: 라운드 실행 메커니즘

### 파일 기반 통신 구조

```
/tmp/debate-{session-id}/
├── topic.md              # 주제 + 메타정보 (복잡도, 형식, 참여자)
├── round-1/
│   ├── prompt.md         # 이번 라운드 프롬프트 (주제 + 이전 맥락)
│   ├── gemini.md         # Gemini 발언
│   ├── codex.md          # Codex 발언
│   └── claude.md         # Claude 발언
├── round-2/
│   ├── prompt.md
│   ├── gemini.md
│   ├── codex.md
│   └── claude.md
└── ...
```

### 라운드 1개의 실행 흐름

1. **prompt.md 작성** (Claude)
   - 주제 설명
   - 이전 라운드 발언 요약 (라운드 2+)
   - 이번 라운드에서 기대하는 것 (형식에 따라 다름)

2. **tmux로 Gemini/Codex 동시 스폰**
   - 각 워커에게 prompt.md + 역할 전달
   - 출력을 각자의 파일(gemini.md, codex.md)에 저장

3. **발언 수집 대기**
   - 양쪽 파일이 모두 생성될 때까지 대기
   - 타임아웃: 워커당 최대 120초

4. **Claude 발언**
   - Gemini/Codex 발언을 읽고 자기 의견 작성
   - 동의/반박/새로운 관점 추가

5. **수렴 판단**
   - 핵심 논점별로 참여자 입장 비교
   - 수렴 기준: 새로운 논점이 없고, 주요 쟁점에서 입장 차이 축소
   - 수렴 or max round 도달 → Step 3
   - 미수렴 → 다음 라운드 prompt에 "미해결 쟁점" 명시

### 라운드별 프롬프트 진화

| 라운드 | 프롬프트 내용 |
|--------|-------------|
| 1 | 주제 제시 + 자유 의견 요청 |
| 2+ | 이전 발언 요약 + 미해결 쟁점 + "이 부분에 대해 반응해 주세요" |
| 마지막 | "최종 입장을 정리해 주세요" |

## 워커 실행 상세

### Gemini/Codex 스폰 방식

omc-teams의 tmux 워커를 사용한다. 각 라운드에서:

```bash
# 1. 세션 디렉토리 생성
mkdir -p /tmp/debate-{session-id}/round-{n}

# 2. prompt.md 작성 후, 각 모델을 tmux 패널로 스폰
# Gemini 워커
tmux new-window -t debate "gemini-cli --prompt-file /tmp/debate-{session-id}/round-{n}/prompt.md > /tmp/debate-{session-id}/round-{n}/gemini.md"

# Codex 워커
tmux split-window -t debate "codex-cli --prompt-file /tmp/debate-{session-id}/round-{n}/prompt.md > /tmp/debate-{session-id}/round-{n}/codex.md"
```

실제 구현에서는 OMC의 `omc_run_team_start` MCP 도구를 사용하여 워커를 스폰한다. 위 bash 예시는 내부 동작의 개념 설명이다.

### 프롬프트 전달 방식

각 워커에게 전달되는 prompt.md 구조:

```markdown
# 토론 주제
{주제}

# 당신의 역할
당신은 이 주제에 대해 독립적으로 의견을 제시하는 토론 참여자입니다.
형식: {라운드테이블|디베이트|소크라테스}
{형식별 추가 지시 — references/format-rules.md에서 로드}

# 이전 라운드 요약 (라운드 2+)
{이전 발언 요약 + 미해결 쟁점}

# 요청
{라운드별 요청 — 1: 자유 의견, 2+: 쟁점 반응, 마지막: 최종 입장}

# 출력 형식
- 핵심 주장 1줄 요약으로 시작
- 근거를 구체적으로 제시
- 500자 내외
```

### 완료 감지

- 워커의 출력 파일(gemini.md, codex.md)이 존재하고 크기 > 0이면 완료로 판단
- 5초 간격으로 폴링
- 타임아웃: 120초 (초과 시 해당 워커 스킵, 나머지 발언으로 진행)

## 실패 처리

| 실패 유형 | 감지 방법 | 복구 행동 |
|-----------|-----------|-----------|
| 워커 타임아웃 (120초) | 파일 미생성 or 크기 0 | 해당 워커 스킵, 나머지 참여자로 라운드 속행 |
| 워커 크래시 (tmux 패널 종료) | tmux 패널 상태 확인 | 1회 재시도 → 실패 시 스킵 |
| 쓰레기 출력 (off-topic) | Claude가 발언 읽고 판단 | "관련 없는 응답" 표시 후 해당 발언 무시 |
| 모든 워커 실패 | 워커 응답 0개 | 토론 중단, 에러 메시지 출력, 임시 파일 정리 |
| 참여자 1명만 남음 | 응답한 워커 수 < 2 | Claude + 1명으로 속행 (최소 2인 필요) |

최소 참여자 수: **2명** (Claude 포함). 1명만 남으면 토론 의미 없으므로 종료.

## 수렴 판단 알고리즘

각 라운드 종료 후 Claude가 다음 절차로 수렴을 판단한다:

### 1. 논점 추출

각 참여자 발언에서 `{논점, 입장}` 쌍을 추출한다.

```
라운드 1:
  Claude:  {일자리 대체: 부분적} {창의 영역: 보완}
  Gemini:  {일자리 대체: 역사적 유사} {신규 직업: 생성}
  Codex:   {전환 속도: 핵심 변수} {재교육: 필수}
```

### 2. 라운드 간 비교 (라운드 2+)

- **새 논점 등장 여부**: 이전 라운드에 없던 논점이 나왔는가?
- **입장 변화**: 기존 논점에서 참여자 입장이 수렴했는가?
- **명시적 동의**: "동의한다", "맞다" 등 명시적 합의 표현

### 3. 수렴 조건

다음 **모두** 충족 시 수렴으로 판단:
- 새로운 논점이 0개
- 기존 논점의 2/3 이상에서 참여자 입장이 일치 또는 조건부 동의

미수렴 시 → 다음 라운드 prompt에 "미해결 쟁점" 목록을 명시.

## 자가평가 루브릭

baseline: 3.5 / 5

| 항목 | 1 | 3 | 5 |
|------|---|---|---|
| **논점 다양성** | 한 가지 관점만 반복 | 2-3개 관점 제시 | 다각도 + 예상 못한 관점 포함 |
| **수렴 품질** | 억지 합의 또는 수렴 실패 | 주요 쟁점에서 자연스러운 합의 | 합의와 이견이 모두 명확하게 구분 |
| **결론 정당성** | 토론 내용과 결론이 불일치 | 토론에서 도출 가능한 결론 | 근거가 명확하고 결론이 필연적 |
| **균형 표현** | 한 모델 의견이 지배적 | 각 모델 비중이 대체로 균등 | 모든 참여자 고유 기여가 명확 |
| **기록 가독성** | 기록만으로 흐름 파악 불가 | 라운드 소결로 흐름 추적 가능 | 두괄식 + 합의표 + 기록이 일관되게 연결 |

frontmatter에 `eval_scores: {논점다양성: N, 수렴품질: N, ...}` 포함.
`insights/_quality/tracker.md`에 등록. 테이블 형식:

```
| 파일명 | 버전 | 논점다양성 | 수렴품질 | 결론정당성 | 균형표현 | 기록가독성 | 평균 | 날짜 |
```

## 모델 레지스트리

| 모델 ID | CLI 도구 | 비고 |

## 기존 시스템 연동

- `insights/debates/` 폴더를 새로 생성 (소스·분석 노트와 분리)
- frontmatter에 `skill: debate`, `skill_version` 포함하여 self-improve 스킬과 호환
- 태그 기반으로 기존 분석 노트와 Obsidian 그래프에서 연결 가능
- `/tmp/debate-{session-id}/` 임시 파일은 노트 저장 후 정리
