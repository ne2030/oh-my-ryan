## 프로토타입 커버리지 계획

초안 프로세스(v2 설계 + e2e-prototype)로 4-5건을 돌리면서 개선, 충분히 검증되면 스킬로 자동화(일반화).

### 테스트 케이스 (4-5건)

| # | 소스 | 입력 형태 | 예상 깊이 | 주요 렌즈 | 커버하는 분기 |
|---|------|-----------|-----------|-----------|-------------|
| 1 | Byeongchan Park — AI 네이티브 개발 (있음) | 텍스트 | B | 기술+프로세스+인지 | 첫 파이프라인 완주 |
| 2 | Kyunghun Lee — SaaSpocalypse (있음) | 텍스트 | B | 조직·문화+인지 | 다른 렌즈 조합 + 태그 겹침 검증 |
| 3 | (사용자 제공 예정) | 스크린샷 | A | 소스에 따라 | 이미지 입력 파싱 + A깊이 |
| 4 | (사용자 제공 예정) | 링크 | C | 소스에 따라 | 링크 페칭 + C깊이(리서치) |
| 5 | (선택, 사용자 제공 예정) | 자유 | 자유 | 자유 | 태그 링킹 밀도 확인 |

**분기 커버리지 목표:**
- 입력 형태: 텍스트(1,2) / 스크린샷(3) / 링크(4) — 3종 완료
- 분석 깊이: A(3) / B(1,2) / C(4) — 3종 완료
- 렌즈: 기술(1) / 프로세스(1) / 조직·문화(2) / 인지(1,2) — 4종 완료
- 태그 링킹: 1-2건 이후 태그 겹침으로 [[위키링크]] 자동 연결 검증

### 진행 방식

```
건 N 시작
  -> 초안 프로세스로 분석 노트 생성 (Step 1-4)
  -> Claude 자체 평가 (5항목 점수)
  -> 사용자에게 제시 + 평가 요청
  -> Good: 건 N+1 진행
  -> Adjust: 해당 건 수정 + 프로세스 메모 업데이트
  -> Rethink: v2 설계 수정 후 재시도
  -> 4-5건 완료 후: 종합 회고 -> 스킬 자동화 설계
```

## 산출물 품질 평가 기준

**핵심 원칙: 산출물 품질이 최우선 평가 기준**

### Claude 자체 평가 (분석 노트 생성 직후, 매 건)

| 항목 | 기준 | 점수 |
|------|------|------|
| **인사이트 유용성** | 원본을 안 읽어도 핵심을 파악할 수 있는가 | 1-5 |
| **계층 분류 자연스러움** | 원칙/사례/프레임 분류가 억지가 아닌가 | 1-5 |
| **실행 가능성** | "나는 이걸 이렇게 적용해야지" 하는 액션이 나오는가 | 1-5 |
| **근거 충분성** (B/C) | 주장이 설득력 있게 뒷받침되는가 | 1-5 |
| **하네스 연결성** | 하네스 적용 제안이 구체적이고 유용한가 | 1-5 |

**기준선**: 평균 3.5 이상이면 합격, 미만이면 프로세스 개선 필요

### 사용자 판단 (간단하게)

- **Good** — 이대로 충분, 다음 건 진행
- **Adjust** — 이 부분 고쳐줘 (구체 피드백) → 해당 건 수정 + 프로세스 반영
- **Rethink** — 접근 자체가 잘못됨 → 프로세스 재설계

### 보조 평가 (프로세스/Obsidian, 4-5건 종합 시)

- 라우팅 추천 정확도: 사용자가 override한 비율
- 렌즈 선택 적절성: 사후에 "이 렌즈도 봤어야 했는데" 비율
- Obsidian 통합: 그래프 뷰에서 연결이 의미 있는가, 태그 검색이 쓸만한가
