# oh-my-ryan v2: 분석 프로세스 상세 설계

> Deep Interview 결과 (12 rounds, ambiguity: 17.5%)

## 1. 분석 프레임워크 개요

소스를 받았을 때 다음 파이프라인으로 처리한다:

```
소스 입력
  -> 소스 성격 파악 (자동)
  -> 분석 깊이 추천 + 사용자 승인 (A/B/C)
  -> 분석 렌즈 자동 선택
  -> 인사이트 계층화 추출
  -> 근거 확보 (깊이에 따라)
  -> 하네스 적용 제안
  -> 태그 기반 기존 노트 링킹
  -> 분석 노트 저장
```

## 2. 분석 렌즈 (Analysis Lenses)

소스 성격에 따라 적합한 렌즈를 자동 선택한다. 하나의 소스에 여러 렌즈가 적용될 수 있다.

| 렌즈 | 설명 | 주요 질문 | 적합한 소스 |
|------|------|-----------|-------------|
| **기술 (Technical)** | 어떤 기술/도구/아키텍처를 쓰는가 | "무엇을 사용하나?", "어떻게 구현하나?" | 기술 블로그, 논문, 튜토리얼 |
| **프로세스 (Process)** | 어떤 워크플로우/방법론으로 일하는가 | "어떤 순서로 하나?", "어떻게 반복하나?" | 경험담, 사례 공유, 워크플로우 소개 |
| **조직/문화 (Org/Culture)** | 어떤 환경/구조가 이것을 가능케 하는가 | "어떤 조직이 필요한가?", "어떤 문화가 뒷받침하나?" | 경영/조직 글, 스타트업 사례 |
| **인지/멘탈모델 (Cognitive)** | 어떤 사고방식/관점 전환이 핵심인가 | "어떤 가정을 깨는가?", "어떤 프레임이 바뀌나?" | 오피니언, 에세이, 통찰 공유 |

### 렌즈 자동 선택 로직

소스의 주요 키워드, 주제, 저자 프로필을 기반으로:
- 기술적 용어(API, 아키텍처, 구현) 많으면 -> 기술 렌즈
- 워크플로우, 프로세스, 단계 설명이 있으면 -> 프로세스 렌즈
- 조직, 팀, 문화, 채용 이야기가 있으면 -> 조직/문화 렌즈
- 사고 전환, "깨달음", 프레임 변화가 있으면 -> 인지 렌즈
- 여러 렌즈 해당 시 복수 적용
- 소스의 일부(10-20%)만 해당하더라도 핵심 인사이트가 있으면 렌즈를 부분 적용으로 포함

## 3. 인사이트 계층화 (Insight Hierarchy)

모든 인사이트를 3계층으로 분류한다:

### 3.1 원칙 (Principles)
- 일반화 가능한 추상적 원리/패턴
- 다른 맥락에도 적용 가능
- 예: "AI의 실수를 시스템으로 잡으면 사람 개입이 줄어든다"

### 3.2 사례 (Cases)
- 구체적인 적용 사례, 수치, 결과
- 특정 맥락에 종속되지만 참고 가치가 있음
- 예: "105K줄 테스트로 규칙 위반 자동 차단"

### 3.3 프레임 (Frames)
- 사고방식/역할/관점의 전환
- 패러다임 시프트나 멘탈모델 변화
- 예: "코드 작성자 -> 의사결정자 + 시스템 설계자"

## 4. A/B/C 깊이별 상세 정의

### 4.1 깊이 A: 소스 분석 (Source-Only)

**언제**: 오피니언, 감상, 짧은 코멘트, 추상적 글
**작업 범위**:
- 소스 텍스트만 보고 분석
- 외부 검색/검증 없음

**산출물**:
- 원본 요약 (3-5문장)
- 핵심 인사이트 (원칙/사례/프레임 계층화)
- 적용된 분석 렌즈 표시
- 관련 노트 링킹 (태그 기반)
- 하네스 적용 제안 (해당 시)

### 4.2 깊이 B: 소스 분석 + 검증 (Source + Verification)

**언제**: 경험 공유, 사례 중심 글, 구체적 주장이 있는 글
**작업 범위**:
- A의 모든 것 +
- 소스 내부 근거 추출 (수치, 데이터, 구체적 사실)
- 주장의 사실 여부 간단 검증 (가벼운 외부 확인)
- 저자/기업 배경 확인

**산출물**:
- A의 모든 것 +
- 근거 및 출처 섹션
  - 내부 근거: 소스 안의 수치/데이터/사실 정리
  - 검증: 주요 주장의 사실 확인 결과

### 4.3 깊이 C: 소스 분석 + 검증 + 리서치 (Source + Verification + Research)

**언제**: 논문, 기술적으로 깊은 글, 새로운 개념/프레임워크 소개
**작업 범위**:
- B의 모든 것 +
- 웹 검색으로 관련 글/논문/블로그 탐색
- context7 등 도구로 기술 문서 참조
- 지지/반박 자료 수집
- 관련 연구/사례 추가 발굴

**산출물**:
- B의 모든 것 +
- 추가 리서치 섹션
  - 관련 자료 목록 (링크 + 요약)
  - 지지/반박 근거
  - 확장된 맥락

## 5. 라우팅 기준 (A/B/C 추천 로직)

소스 성격에 따라 자동 추천하되, 반드시 사용자 승인을 받는다.

| 소스 성격 | 추천 깊이 | 판단 근거 |
|-----------|-----------|-----------|
| 짧은 코멘트, 트윗, 감상 | A | 검증할 주장이 적음 |
| 오피니언, 에세이 | A | 주관적 관점 중심 |
| 경험 공유, 사례 글 | B | 구체적 수치/사례가 있어 검증 가치 있음 |
| 기술 블로그 (코드 포함) | B | 기술적 정확성 검증 필요 |
| 논문, 기술 deep-dive | C | 깊은 맥락 파악이 가치를 높임 |
| 새로운 개념/프레임워크 소개 | C | 기존 연구/사례와의 연결이 중요 |
| 시장/트렌드 분석 | B-C | 데이터 검증 + 추가 자료로 보강 |

### 추천 프롬프트 형태

```
이 소스를 분석하겠습니다.

소스 유형: LinkedIn 경험 공유
분석 렌즈: 기술 + 프로세스
추천 깊이: B (소스에 구체적 수치와 사례가 있어 검증 가치 있음)

B로 진행할까요? (A: 가볍게 / B: 검증 포함 / C: 리서치까지)
```

## 6. 근거 확보 방법

### 6.1 내부 근거 (모든 깊이)

소스 텍스트 안에서 추출:
- 정량적 데이터 (숫자, 비율, 기간)
- 구체적 사례 (실제 적용 결과)
- 논리적 추론 체인 (A이므로 B, B이므로 C)

### 6.2 간단 검증 (B 깊이)

- 저자/기업 프로필 확인 (주장의 신뢰성)
- 언급된 기술/도구의 실제 존재 여부
  - 수치의 합리성 검토 (상식적으로 가능한 범위인가)

### 6.3 본격 리서치 (C 깊이) 

- 웹 검색: 관련 블로그, 뉴스, 글
- 기술 문서: context7로 라이브러리/프레임워크 문서 참조
- 논문: 학술적 근거가 있다면 연결
- 반박 자료: 해당 주장에 대한 다른 관점/비판

## 7. 하네스 적용 제안 형태

인사이트에서 하네스에 적용 가능한 부분이 있을 때 자동 제안한다.

형태:
```markdown
## 하네스 적용 제안

- **적용 영역**: [하네스의 어떤 모듈/영역]
- **방향**: [어떤 개념을 어떻게 적용할 수 있는지 1-2문장]
- **참고**: [[관련 분석 노트 링크]]
```

하네스가 아직 초기 단계이므로, 적용 영역은 넓게 잡되 구체적 실행보다는 방향 제시에 집중.
하네스가 구체화됨에 따라 이 섹션도 더 구체적으로 진화.

## 9. 품질 기준

분석 노트가 성공적이려면:

1. **깊은 이해**: 원본 글보다 맥락을 더 잘 이해할 수 있어야 함
2. **실행 가능한 테이크아웃**: 읽고 나서 "나는 이걸 이렇게 적용해봐야지" 하는 구체적 액션이 나와야 함
3. **계층화된 인사이트**: 원칙/사례/프레임이 명확히 구분되어야 함
4. **근거 기반** (B/C): 주장이 뒷받침되어야 함
5. **연결성**: 기존 노트와 태그로 연결되어 그래프에서 의미 있는 노드가 되어야 함
