# 분석 가이드: 요약, 계층화, 검증

## 1. 원본 요약 작성법

3-5문장으로 작성하되, 아래 4요소를 반드시 포함한다:

| 요소 | 설명 | 예시 |
|------|------|------|
| **누가** | 저자 + 맥락 (직책, 소속) | "채널톡의 AI Native Engineer인 Park이" |
| **무엇을** | 핵심 주장 또는 경험 | "6개월간 코드 0줄로 560개 PR을 머지한 경험을 공유한다" |
| **핵심 근거** | 주장을 뒷받침하는 가장 강력한 데이터/논거 1-2개 | "105K줄 테스트, CI 10분→2분 단축" |
| **결론/시사점** | so what? 이것이 왜 중요한가 | "AI Native 레포를 만드는 것은 좋은 엔지니어링 그 자체다" |

**금지 사항**:
- "~에 대해 이야기한다" 같은 메타 서술로 끝내지 않기
- 저자의 결론을 반드시 포함할 것 (요약만 하고 결론 누락 금지)

## 2. 인사이트 계층화 기준

### 원칙 (Principles) — "다른 맥락에서도 성립하는가?"

- **판별**: 소스의 구체적 맥락(회사, 기술, 시점)을 제거해도 여전히 유효한 명제
- **형태**: "~하면 ~한다" 또는 "~보다 ~가 중요하다" 형태의 일반 법칙
- **예**: "시스템이 AI 실수를 잡으면 사람 개입이 줄어든다" → 채널톡이 아니라 어디서든 성립
- **개수**: 2-4개. 너무 많으면 추상화가 부족한 것

### 사례 (Cases) — "구체적 수치/결과가 있는가?"

- **판별**: 특정 맥락에 종속된 구체적 적용 결과
- **필수 포함**: 수치, 기간, 규모, 결과 중 최소 1개
- **형태**: "[주체]가 [행동]하여 [결과]를 달성" 형태
- **예**: "22개 레포에서 560+ PR, 최대 10개 동시 작업" → 채널톡의 구체적 수치
- **개수**: 3-6개. 소스 내 주요 데이터 포인트를 모두 포착

### 프레임 (Frames) — "기존 사고방식을 깨는가?"

- **판별**: 기존 가정 A를 새로운 관점 B로 전환하는 명제
- **필수 형태**: "A → B" (기존 → 새로운) 전환 구조
- **원칙과의 차이**: 원칙은 "무엇이 참인가"를 말하고, 프레임은 "무엇이 바뀌었는가"를 말한다
- **예**: "코드 작성자 → 의사결정자" → 역할의 전환을 표현
- **개수**: 1-3개. 없을 수도 있음 (모든 소스에 프레임 전환이 있지는 않다)

### 경계가 모호할 때

| 상황 | 판정 |
|------|------|
| 맥락 제거해도 성립 + 전환 구조 없음 | 원칙 |
| 맥락 제거해도 성립 + "A→B" 전환 구조 있음 | 프레임 |
| 구체적 수치/결과 포함 | 사례 |
| 동일 내용이 원칙이자 프레임 같음 | 프레임 우선 (전환이 더 강한 인사이트) |

## 3. 검증 방법 (B 깊이)

### 3.1 내부 근거 추출

소스 텍스트에서 아래를 테이블로 정리한다:

| 추출 대상 | 설명 |
|-----------|------|
| 정량 데이터 | 숫자, 비율, 기간, 금액 |
| 구체적 사례 | 실제 적용 결과, 전후 비교 |
| 논리 체인 | "A이므로 B, B이므로 C" 형태의 추론 |

### 3.2 외부 검증 — URL 필수

검증한 내용에는 **반드시 출처 URL 또는 출처명을 명시**한다. "확인됨"이라고만 쓰지 않는다.

검증 항목별 방법:

| 검증 대상 | 방법 | 출처 표기 예시 |
|-----------|------|---------------|
| **저자/기업 프로필** | LinkedIn, 기업 웹사이트 확인 | `LinkedIn 프로필 (https://linkedin.com/in/...)` |
| **재무 데이터** | 실적 발표, IR 자료, 뉴스 기사 | `Workday FY2026 Q3 Earnings (URL)` |
| **제품/기술 존재 여부** | 공식 문서, 릴리즈 노트 | `UiPath 공식 뉴스룸 (URL)` |
| **주장의 합리성** | 업계 벤치마크, 유사 사례 대조 | 상식적 범위 판단은 URL 불필요, 근거 논리만 서술 |

### 3.3 검증 불가 시

검증할 수 없는 주장은 **명시적으로 "미검증"으로 표기**한다:
```
- **[주장 내용]**: 미검증. [검증 불가 사유: 비공개 데이터 / 개인 경험 / 출처 불명]
```

## 4. 검증 방법 (C 깊이 — B에 추가)

B의 모든 검증 수행 후 추가로:

### 4.1 관련 자료 탐색
- WebSearch로 핵심 키워드 검색 (최소 3회)
- 검색어 예: "[핵심개념] research", "[저자명] [주제]", "[기술명] benchmark"
- context7로 기술 문서 참조 (기술 렌즈 적용 시)

### 4.2 지지/반박 자료 수집
- 동일 주장을 지지하는 다른 출처 최소 1개
- 반박하거나 다른 관점을 제시하는 출처 최소 1개 (있는 경우)
- 각 자료: `[제목](URL) — 1문장 요약` 형태

### 4.3 확장된 맥락
- 이 소스가 속한 더 큰 담론/트렌드는 무엇인가
- 시간축에서 이 소스의 위치 (선행 연구 → 이 소스 → 후속 발전)

## 5. 분석자 코멘트

"관련 노트" 앞, "근거 및 출처" 뒤에 `## 분석자 코멘트` 섹션을 작성한다.

### 5.1 논리 평가

이 글의 논증 구조를 평가한다:

- **논증 구조**: 주장 → 근거 → 결론의 흐름이 명확한가?
- **약한 고리**: 논리적 비약, 근거 없는 일반화, 생존자 편향 등이 있는가?
- **설득력**: 1-5점 (1=주장만, 5=데이터+논리+반론 고려)

예시:
```
- **논증 구조**: 실패 경험(RAG 기법 총동원) → 유일한 성공(데이터 클렌징) → 결론(병목은 데이터). 귀납적 논증.
- **약한 고리**: N=1. 본인 경험만으로 "유일한 방법"이라 일반화. 다른 기업/도메인에서는 다를 수 있음.
- **설득력**: 3/5 — 체험은 생생하나 비교 실험 설계가 아님.
```

### 5.2 신뢰도 판정

검증 결과를 종합하여 판정한다:

| 등급 | 기준 |
|------|------|
| **높음** | 핵심 주장의 80%+ 외부 검증 가능, 저자 신뢰성 확인 |
| **중간** | 일부 검증 가능, 일부 미검증이나 합리적 범위 |
| **낮음** | 대부분 미검증, 또는 검증 결과 사실과 불일치 |

A깊이(외부 검증 없음)에서는 "신뢰도 판정 불가 — A깊이"로 표기.

### 5.3 확장 설명

소스에서 사용된 기술 개념/용어 중 추가 설명이 가치 있는 것을 골라서 풀어 설명한다.

**선택 기준**:
- 소스가 당연하다는 듯 사용하지만 배경 지식 없이는 이해 어려운 개념
- 소스가 간략히 언급했지만 더 깊이 알면 인사이트가 확장되는 개념

**형태**:
```
> **pid 기반 tmp폴더 격리**: Unix에서 각 프로세스는 고유한 Process ID(pid)를 가진다.
> tmp폴더명에 pid를 포함시키면(예: /tmp/agent-12345/) 동시에 여러 프로세스가 돌아도
> 폴더가 겹치지 않는다. 이는 컨테이너 없이도 프로세스 수준의 격리를 달성하는 가장 단순한 방법.
```

모든 개념을 설명하지 않는다. 소스의 핵심 인사이트 이해에 필요한 것만 선별.

### 5.4 비판적 관점

이 글이 놓친 것, 반대 의견, 한계를 제시한다:

- **놓친 관점**: 저자가 다루지 않은 중요한 측면
- **반대 의견**: 이 주장에 대한 알려진 반론
- **한계/전제 조건**: 이 인사이트가 성립하는 조건과 성립하지 않는 조건

비판이 없으면 억지로 만들지 않는다. 다만 모든 글에 한계는 있으므로 최소 "한계" 1개는 제시.

**근거 필수 원칙 — 비판에도 동일하게 적용**:
- 수치를 인용하면 **반드시 출처를 명시**한다 (예: "구글 검색 점유율 90%" → 출처 URL 또는 출처명 필수)
- 출처를 찾을 수 없는 수치는 사용하지 않는다. 대신 논리적 근거로 비판한다
- "~로 알려져 있다", "일반적으로 ~하다" 같은 표현도 근거 없이 쓰지 않는다
- **비판의 신뢰도는 비판 대상보다 높아야 한다** — 근거 없는 비판은 분석 노트 전체의 신뢰도를 떨어뜨린다

```
# 나쁜 예
- **한계**: AI 검색이 전체 검색 트래픽에서 차지하는 비중은 아직 매우 낮음 — 구글이 여전히 검색 시장의 90%+ 점유.

# 좋은 예
- **한계**: AI 검색이 전체 검색 트래픽에서 차지하는 비중은 아직 제한적 — StatCounter GlobalStats(2025.02) 기준 구글 검색 점유율 89.3%.

# 출처를 못 찾을 때
- **한계**: AI 검색의 실질적 시장 영향력은 아직 초기 단계로, 주류 검색 행동을 대체했다는 데이터가 부족하다.
```

## 6. 하네스 적용 제안

### 하네스란?

하네스 = LLM(Claude Code, Codex 등)을 활용하여 원하는 결과를 만들어내는 **개인 환경/툴/워크플로우 전체**.

이 프로젝트(oh-my-ryan) 하나에 국한되지 않는다. 하네스에서 나올 수 있는 것들:
- 딥 리서치 툴, 마케팅 팀, devil's advocate 같은 방법론 스킬
- oh-my-claudecode 같은 큰 프레임워크
- Claude Code 관점에서 마켓플레이스 + 플러그인 생태계

**인사이트 중 하네스에 적용 가능한 것이 있을 때만 작성한다.**

**작성 기준**:
- 하네스의 현재 구체화 수준에 맞춰 작성 (초기: 방향 제시, 구체화 후: 모듈/기능 지정)
- "적용 가능할 수 있다" 같은 모호한 제안 금지
- 적용 영역 + 구체적 방향 + 소스 참조를 반드시 포함
- 해당하지 않으면 섹션 자체를 비워둔다 (억지 연결 금지)
