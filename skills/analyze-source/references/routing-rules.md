# 라우팅 규칙: 깊이 추천 + 렌즈 선택

## 깊이 추천 (A/B/C)

소스의 성격을 파악하고 아래 표에 따라 깊이를 추천한다.

| 소스 성격 | 추천 깊이 | 판단 근거 |
|-----------|-----------|-----------|
| 짧은 코멘트, 트윗, 감상 (500자 미만) | A | 검증할 구체적 주장이 없음 |
| 오피니언, 에세이, 주관적 관점 중심 | A | 사실 검증보다 프레임 추출이 가치 |
| 경험 공유, 사례 글 (수치/데이터 포함) | B | 구체적 수치가 있어 검증 가치 있음 |
| 기술 블로그 (코드/아키텍처 포함) | B | 기술적 정확성 검증 필요 |
| 시장/트렌드 분석 (재무 데이터 포함) | B | 데이터 교차 검증 필요 |
| 논문, 기술 deep-dive | C | 관련 연구/사례 연결이 가치를 높임 |
| 새로운 개념/프레임워크 소개 | C | 기존 연구와의 맥락 파악이 필수 |

### 복합 판단

- 소스가 여러 성격을 가질 때: 가장 높은 깊이를 추천
- 시장 분석 + 구체적 재무 데이터 → B (데이터가 공개 검증 가능하므로)
- 시장 분석 + 새로운 프레임워크 제안 → C (프레임워크 검증에 리서치 필요)

## 렌즈 선택

### 4개 렌즈와 트리거 조건

#### 기술 (Technical)
- **핵심 질문**: "무엇을 사용하나?", "어떻게 구현하나?"
- **트리거 키워드**: API, 아키텍처, 구현, 코드, 시스템, 스택, 프레임워크, 인프라, CI/CD, 테스트
- **적합한 소스**: 기술 블로그, 논문, 튜토리얼, 구현 사례

#### 프로세스 (Process)
- **핵심 질문**: "어떤 순서로 하나?", "어떻게 반복하나?"
- **트리거 키워드**: 워크플로우, 프로세스, 단계, 방법론, 파이프라인, 자동화, 루프, 반복
- **적합한 소스**: 경험담, 사례 공유, 워크플로우 소개

#### 조직/문화 (Org/Culture)
- **핵심 질문**: "어떤 환경/구조가 이것을 가능케 하는가?"
- **트리거 키워드**: 조직, 팀, 문화, 채용, 경영, CEO, 이사회, 리더십, 스케일링, 의사결정
- **적합한 소스**: 경영/조직 글, 스타트업 사례, 시장 분석

#### 인지/멘탈모델 (Cognitive)
- **핵심 질문**: "어떤 가정을 깨는가?", "어떤 프레임이 바뀌나?"
- **트리거 키워드**: 깨달음, 관점, 전환, 패러다임, 프레임, 사고방식, 가정, 재정의
- **적합한 소스**: 오피니언, 에세이, 통찰 공유

### 선택 규칙

1. **복수 적용이 기본**: 대부분의 소스는 2-3개 렌즈에 해당한다
2. **부분 적용 허용**: 소스의 10-20%만 해당하더라도 핵심 인사이트가 있으면 "(부분)" 표시로 포함
3. **렌즈 선택 근거 필수**: 각 렌즈마다 소스 내 해당 키워드/문장을 최소 1개 인용
4. **주력 렌즈 표시**: 가장 비중이 큰 렌즈를 첫 번째로 배치

### 근거 제시 형태

```
- 렌즈 선택 근거:
  - 기술(Technical): "105K줄 테스트", "CI 10분→2분", "프리커밋 훅" ← 소스 내 실제 표현 인용
  - 프로세스(Process): "병렬 실행 전략", "2개→10개 점진적 확장"
  - 인지(Cognitive) (부분): "코드 작성자 → 의사결정자" ← 마지막 문단에서만 등장하지만 핵심적
```
